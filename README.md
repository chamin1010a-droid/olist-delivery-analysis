브라질 이커머스 플랫폼 **Olist**의 주문 99,441건(2016–2018)을 분석해, 배송 지연이 고객 만족도에 미치는 영향을 규명하고 **예상 도착일 예측 모델**로 개선 가능성을 실증한 프로젝트입니다.

> **Stack** · SQLite + SQL · LightGBM(분위 회귀) · pandas / scikit-learn
> **Data** · [Brazilian E-Commerce Public Dataset by Olist (Kaggle)](https://www.kaggle.com/datasets/olistbr/brazilian-ecommerce)

## 프로젝트 배경 · 접근 방식

이 프로젝트는 *"SQL을 연습하자"*에서 출발하지 않았습니다. **"AI가 SQL을 다 짜주는 시대에, 분석가가 실제로 갖춰야 할 능력은 무엇인가?"** 라는 질문에서 시작했습니다. 문법 암기나 쿼리 작성은 도구가 대신할 수 있습니다. 대신할 수 없는 것은 **어떤 문제를 풀 가치가 있는지 정의하고, 언제 어떤 도구가 정답인지 판단하는 일**이라고 보고, 그 능력을 기르는 연습으로 이 분석을 설계했습니다.

그래서 도구가 아니라 **질문을 먼저** 세웠습니다 — *"내가 Olist의 데이터 분석가로 고용됐고, 사장이 '어떻게 성장시킬까'를 묻는다면 데이터로 무엇을 답할 것인가?"* 이 프레이밍에서 출발해, 고객 불만이 가장 날카롭게 집중되는 지점인 **배송 경험**으로 문제를 좁혔습니다. 분석 과정 내내 SQL과 LightGBM은 *결론을 내리기 위한 수단*으로만 등장하며, 매 단계는 "이걸 알아내고 싶다 → 그래서 이 도구를 쓴다" 순서로 진행됩니다.

분석의 뼈대는 **`SQL EDA → 원인 분리 → ML로 개선 실증 → 정책 제안`** 4단계 파이프라인입니다. 이 파이프라인은 의도적으로 재사용 가능하게 설계했으며, 다른 도메인에도 그대로 적용했습니다 (→ [분석 시리즈](#분석-시리즈)).

## 핵심 결과 한눈에

| 발견 | 수치 |
|------|------|
| **하루만 늦어도 리뷰가 절벽 낙하** | 정시 4.07점 → 1\~7일 늦음 **2.77점**, 7일+ 늦으면 약 70%가 1점 |
| **"빨리 오는 건 보너스가 아니다"** | 15일 일찍(4.32) ≈ 7일 일찍(4.31) — 조기 도착은 만족도에 거의 무영향 |
| **지연의 주범은 물류 운송** | 정상 7.4일 → 지연 시 **36.1일**(약 5배). 판매자 출고 지연은 2배 수준 |
| **예측이 과도하게 보수적** | 전체 주문의 **90.5%가 예상보다 일찍 도착**(평균 −11일) |
| **예측 모델로 오차 45% 감소** | 평균 오차 13.3일 → **7.3일**, 15일+ 과대예측 37.3% → **3.7%(1/10)** |

## 분석 흐름

정해진 단계를 밟은 게 아니라, **한 발견이 다음 질문을 강제하는 식**으로 좁혀졌습니다.

**1. 격차부터 본다 → "너무 보수적인 예측"**
예상 vs 실제 도착일 격차를 7구간으로 나누니, 전체의 **90.5%가 예상보다 일찍**, 평균 11일이나 일찍 도착했습니다. 첫인상은 "예측이 과하게 길다." 그럼 그냥 예상일을 당기면 될까? — 그 전에 **늦음과 일찍 옴 중 무엇이 진짜 문제인지**부터 확인해야 했습니다.

**2. 고객이 민감한 부분**
배송 차이와 리뷰를 붙이니 명확했습니다. **일찍 오는 건 보너스가 아니고**(15일 일찍 4.32 ≈ 7일 일찍 4.31), **하루라도 늦으면 4.07 → 2.77로 절벽**(7일+ 늦으면 약 70%가 1점). 즉 *늦음=재앙, 일찍=거의 무료.* 이 비대칭이 이후 모든 결정의 기준이 됩니다. 할 일이 둘로 갈렸습니다 — **(A) 실제 늦음을 줄인다, (B) 과대예측을 고치되 늦지 않게.**

**3. (A) 늦음은 어디서 오나 → 판매자 vs 물류 분리.**
배송을 *판매자 출고*와 *물류 운송*으로 쪼개니, 지연이 심할수록 **물류가 압도적**이었습니다(정상 7.4일 → 지연 36.1일, 약 5배). 판매자 출고는 2배 수준. → 늦음은 *대부분 물류 문제*다.

**4. 그 늦음은 흩어졌나 몰렸나 → 취약 지점 식별.**
도착지별로 보니 **AL·MA·SE·PI(북동부)는 어디서 출발하든 지연율이 높았습니다** — 출발지가 아닌 *도착지 인프라* 문제. 한편 판매자 출고는 *카테고리*가 좌우하고 고객 지역과는 무관(변동폭 7.1일 vs 0.75일)이라, **카테고리별 기준선**으로 느린 판매자를 콕 집을 수 있었습니다. → 손볼 지점이 구체화됨.

**5. (B) 과대예측은 왜 생기나 → 예측이 원인임을 확인.**
일찍 온 주문들은 실제 배송 속도가 거의 일정(10~14일)한데 **예상일만 길게**(먼 지역일수록 심함 — RO지역은 19일이나 과대) 잡혀 있었습니다. 빠른 게 아니라 *예측이 과한 것.* → 더 정확히 예측할 수 있을까? 단, **2번의 비대칭 때문에 "평균을 맞히는" 예측은 금물**(절반이 늦어버림). 이 제약이 모델 선택을 결정합니다.

> 정리하면: *예측이 보수적이다 → 근데 늦음이 치명적이다 → 늦음은 물류 탓이고 특정 지역에 몰린다 → 과대예측은 예측 자체가 원인인데, 비대칭 탓에 함부로 줄이면 안 된다* — 여기까지 좁힌 뒤 아래 결론에 도달합니다.

## 예측 모델 성능

| 지표 | Olist 현행 | LightGBM 분위회귀 | 변화 |
|------|-----------|-----------------|------|
| MAE (평균 오차) | 13.33일 | **7.28일** | **−45%** |
| 15일+ 과대예측 | 37.3% | **3.7%** | **1/10** |
| 지연율 | 5.1% | 7.9% | +2.8%p |

> 데이터 누수를 피하기 위해 **주문 시점에 알 수 있는 정보만** 사용하고, 시간순 80:20 분할로 검증했습니다. 21일+ 장거리 주문(3.5%)은 표본 부족으로 예측이 불안정해, 경로별 수동 보정 규칙을 함께 제안합니다.

## 결론 · 제안 — 그래서 무엇을 할 것인가

흐름이 도달한 지점은 이것입니다: **늦음은 (대부분 물류발) 치명적 비용이고, 과대예측은 예측 자체가 원인이지만 비대칭 때문에 함부로 줄일 수 없다.** 여기서 결정이 갈립니다.

### 정확도가 아니라 "어느 쪽으로 틀릴지"를 고른다

평균을 맞히는 일반 회귀는 답이 아닙니다(절반이 늦음). 대신 **분위회귀로 85번째 백분위**를 예측했습니다 — "85% 확률로 이 안에 도착", 즉 *의도적으로 늦지 않는 쪽으로 띄운* 예측입니다.

- 평균 오차(MAE) 13.3 → **7.3일 (−45%)**
- 과대예측(15일+ 빗나감) 37.3% → **3.7% (1/10)**
- 지연율 5.1% → 7.9% (비대칭을 감안해 *받아들인* 트레이드오프)

정확도를 올리되, 남는 오차는 비용이 싼 방향(약간 일찍)으로 몰아넣은 것입니다.

### 제안 — 세 갈래, 그리고 한 가지 주의

1. **판매자 관리 체계** — 카테고리별 정상 출고 기준선을 세우고, 크게 초과하는 판매자(예: 콘솔 카테고리 평균 4일인데 18일)를 모니터링·관리합니다.
2. **물류 취약 지역 개선** — AL·MA·SE·PI는 *도착지 인프라* 문제. 신규 물류 허브·지역 물류사 추가를 검토하되, 해당 지역 **매출 규모·성장세와 함께** 투자 대비 효과로 판단합니다.
3. **예상 도착일 알고리즘 교체** — 위 분위회귀로 교체합니다. 단 21일+ 장거리 주문(3.5%)은 표본 부족으로 불안정하니, 경로별 수동 보정 + 운영자 검토를 병행합니다.

> ⚠️ **예측 교체만으로는 실제 배송이 빨라지지 않습니다.** 고객 기대치를 정확히 *관리*하는 조치일 뿐, 근본 속도 개선은 제안 1·2의 몫입니다 — 이 둘을 혼동하지 않는 게 핵심입니다.

## 분석 시리즈

동일한 **`SQL EDA → 원인 분리 → ML로 실증 → 정책 제안`** 파이프라인을 서로 다른 도메인에 적용한 연작입니다. 도메인이 바뀌어도 *문제를 먼저 정의하고 도구는 나중에 끌어온다*는 접근은 동일합니다.

| | 프로젝트 | 도메인 | 차별 포인트 |
|---|---------|--------|------------|
| **Part 1** | **Olist 배송 분석** *(현재 저장소)* | 이커머스 | 파이프라인 원형. 지연 원인을 *판매자 vs 물류*로 분리 |
| **Part 2** | [병원 재입원 분석](https://github.com/chamin1010a-droid/hospital-readmission-analysis) | 헬스케어 | *잔차 분석 · SHAP*로 "환자 중증도 vs 병원 부주의" 인과 분리 |
| **Part 3** | [직원 이직 분석](https://github.com/chamin1010a-droid/employee-attrition-analysis) | HR | 데이터가 작아 *ML 대신 고전 통계*를 택한 도구 판단 자체가 분석 |

> Part 1에서 "예상 도착일 − 실제 도착일" 잔차로 *과대예측 vs 지연*을 분리한 발상을, Part 2에서는 "중증도 예측 − 실제 재입원" 잔차로 *불가피한 재입원 vs 예상 밖 재입원*을 분리하는 데 그대로 확장했습니다.
